# 5.4.2 收集反馈：让使用者告诉你

经过本节学习，你将掌握：
- 为什么反馈比自己闷头想更有用
- 三种零门槛的反馈收集方法
- 一个简单的反馈记录模板
- 如何让 AI 帮你分析反馈

## 为什么需要别人的反馈

你做的工具，你自己觉得好用，不代表别人觉得好用。

根据 ProdPad 2025 年的调研，**91% 的用户认为产品创新应该来自客户的声音**。换句话说，闭门造车做出来的东西，大概率不是用户真正想要的。

你可能会想："我的待办清单就是给自己用的，不需要别人的意见吧？"

其实不然。就算是给自己用，让朋友试一下也能发现问题：
- 你觉得"显而易见"的操作，别人可能根本不知道怎么用
- 你习惯的界面布局，换个人可能觉得很别扭
- 你从来不用的功能，别人可能每天都想用

**找几个人试用，是发现问题最快的方法。**

## 方法一：真人测试法

最简单的方法：把你的待办清单发给 3-5 个人试用。

可以是：
- 家人（特别是不太懂技术的）
- 室友或同事
- 朋友

不需要多，3个人就够了。

### 怎么发给他们

如果你已经完成了 5.2 节的部署，直接把网址发给他们就行：

> "我做了一个待办清单工具，你能帮我试用一下吗？大概花2分钟。网址是 xxx.zeabur.app"

如果还没部署，可以让他们到你电脑前试用。

## 方法二：观察法

观察他们怎么用，比直接问更有用。

当别人在试用时，注意观察：

| 观察点 | 可能说明的问题 |
|--------|--------------|
| 停顿了很久，不知道点哪里 | 界面不够直观 |
| 问你"这个按钮是干嘛的" | 按钮文字不够清楚 |
| 反复点同一个地方 | 交互反馈不明显 |
| 皱眉头、叹气 | 某个操作让他们感到挫败 |
| "哦，原来是这样" | 操作逻辑不符合直觉 |

**不要在旁边指导**。让他们自己摸索，你只是观察。这样才能看到真实的使用情况。

## 方法三：直接问

观察完之后，可以问他们三个问题：

```
1. 第一次打开时，你知道该怎么用吗？
2. 有什么地方让你感到困惑或不方便？
3. 如果可以加一个功能，你希望是什么？
```

这三个问题覆盖了：
- 易用性（能不能快速上手）
- 痛点（哪里不好用）
- 需求（希望有什么功能）

## 记录反馈的模板

收集到的反馈，用一个简单的表格记录下来：

| 反馈来源 | 问题/建议 | 优先级 | 状态 |
|---------|----------|-------|------|
| 室友 | 想要任务截止日期 | P1 | 待开发 |
| 妈妈 | 按钮太小，不好点 | P0 | 已修复 |
| 自己 | 想要分类标签 | P2 | 计划中 |
| 同事 | 能不能有深色模式 | P2 | 计划中 |

### 优先级怎么定

| 优先级 | 含义 | 举例 |
|--------|------|------|
| **P0** | 必须马上修 | 功能坏了、按钮点不动、界面错乱 |
| **P1** | 尽快做 | 影响日常使用的问题，多人反馈的需求 |
| **P2** | 有空再说 | 锦上添花的功能，个别人的特殊需求 |

一个简单的判断标准：
- **影响所有人** → P0 或 P1
- **只影响部分人** → P1 或 P2
- **没人提但你自己想做** → 先放一放

## 让 AI 帮你分析反馈

如果你收集了不少反馈，可以让 AI 帮你整理和分析。

```
我做了一个待办清单工具，收集到了一些用户反馈，请帮我分析：

反馈列表：
1. 室友说想要任务截止日期提醒
2. 妈妈说按钮太小，在手机上不好点
3. 同事说希望有深色模式
4. 我自己觉得任务多了之后不好找，想要分类功能
5. 室友还说任务完成后的动画挺好看的（正面反馈）

请帮我：
1. 按优先级排序这些反馈
2. 分析哪些是共性问题，哪些是个人偏好
3. 建议我先做哪个改进
```

AI 会帮你梳理出一个清晰的行动计划。

## 常见误区

### 只听表扬，忽略批评

"挺好的"、"还不错"这种反馈虽然听着舒服，但对改进没有帮助。

**真正有价值的反馈往往是批评**：
- "我不知道怎么删除任务"
- "这个颜色看着有点刺眼"
- "能不能加个搜索功能"

这些才是改进的方向。

### 试图满足所有人

不同的人会提出不同甚至矛盾的需求。你不需要满足所有人。

回到第二章的灵魂三问：**你的工具到底是给谁用的？** 优先满足核心用户的需求。
