# 3.4.2 从第二章成果到 PRD：整合你的思考

如果你认真完成了第二章的练习，恭喜你——你已经拥有了写 PRD 所需的所有素材。

本节的任务是：**把这些思考成果"搬"到 PRD 的对应位置。**

## 第二章成果 → PRD 字段映射表

| 第二章工具 | 你已完成的思考 | 对应 PRD 字段 |
|-----------|--------------|--------------|
| 灵魂三问 2.5.2 | 用户是谁 | **目标用户** |
| 灵魂三问 2.5.3 | 痛点在哪 | **要解决的问题** |
| 灵魂三问 2.5.4 | 为什么选你 | **一句话描述**（差异化） |
| 减法思维 2.3 | P0 功能列表 | **核心功能** |
| 减法思维 2.3.5 | "不做清单" | **不做的功能** |
| 用户旅程 2.4.4 | 关键步骤 | **用户流程** |
| 故事思维 2.4 | 用户画像 | 融入**目标用户**描述 |

**核心洞见**：PRD 不是从零开始写，而是把第二章的思考"翻译"成 AI 能执行的格式。

## 小李的映射示例

让我们看看小李在第二章做了哪些思考，以及如何填入 PRD：

### 第二章：灵魂三问的答案

```
用户是谁？
→ 我自己，28岁职场人士，每天处理5-10件事

痛点在哪？
→ 怕遗漏重要事项，便签纸容易丢，手机备忘录不够直观

为什么选我的方案？
→ 比便签纸更不容易丢，比复杂待办App更简单容易坚持
```

### 第二章：减法思维的成果

```
P0（必须有）：添加任务、查看列表、标记完成
P1（应该有）：删除任务、任务计数
P2（可以有）：分类标签、优先级标记
不做清单：登录注册、云同步、团队协作、日历视图
```

### 第二章：用户旅程的关键步骤

```
1. 打开网页 → 看到任务列表
2. 输入新任务 → 点击添加
3. 完成任务 → 点击勾选
4. 关闭网页 → 下次打开数据还在
```

### 填入 PRD 后

```markdown
## 目标用户
我自己（28岁职场人士，每天处理5-10件事）

## 要解决的问题
怕遗漏重要事项。便签纸容易丢，手机备忘录不够直观，
需要一个"打开就能用、关掉数据还在"的极简工具。

## 一句话描述
一个比便签纸更持久、比复杂App更简单的待办清单网页

## 核心功能（P0）
- 添加任务
- 查看任务列表
- 标记任务完成

## 不做的功能
- 登录注册
- 云端同步
- 团队协作
- 日历视图
- 分类标签（P2，暂不做）

## 用户流程
1. 打开网页 → 看到任务列表
2. 输入任务 → 点击添加按钮
3. 完成任务 → 点击勾选
4. 关闭网页 → 下次打开数据还在
```

看到了吗？PRD 的每一部分都有来源，不需要凭空编造。

## 填写顺序建议

不是所有字段都同等重要。推荐的填写顺序：

| 顺序 | 字段 | 原因 |
|-----|------|------|
| 1 | 要解决的问题 | 这是核心，决定其他所有内容 |
| 2 | 目标用户 | 问题是谁的问题？ |
| 3 | 核心功能 | 解决问题需要什么功能？ |
| 4 | 不做的功能 | 同样重要，防止范围蔓延 |
| 5 | 用户流程 | 用户怎么用这些功能？ |
| 6 | 技术约束 | 有什么限制条件？ |
| 7 | 成功标准 | 怎么算做完了？ |
| 8 | 一句话描述 | 最后总结提炼 |

**为什么"要解决的问题"排第一？**

因为功能是为问题服务的。如果你先写功能，很容易陷入"我想要这个、那个"的功能堆砌。先写问题，每个功能都要回答："这个功能是在解决什么问题？"

## 常见错误

### 错误一：跳过问题，直接写功能

❌ 错误示范：
```markdown
## 核心功能
- 添加任务
- 删除任务
- 分类标签
- 日历视图
- 提醒通知
```

问题：这些功能在解决什么问题？为什么需要日历视图？

✅ 正确做法：
```markdown
## 要解决的问题
怕遗漏重要事项

## 核心功能
- 添加任务（记录要做的事）
- 查看列表（知道还有什么没做）
- 标记完成（获得成就感）
```

### 错误二：把"不做清单"当摆设

很多人写 PRD 时，"不做清单"要么空着，要么随便填几个。

但这恰恰是 AI 最需要知道的信息。没有明确的边界，AI 会"贴心地"帮你加上各种功能。

✅ 好的"不做清单"：
```markdown
## 不做的功能
- 不要登录注册（这是个人工具）
- 不要云同步（本地存储够用）
- 不要标签分类（保持极简）
- 不要优先级标记（第一版不需要）
- 不要移动端适配（先做桌面版）
```

### 错误三：用户画像太详细

第二章的三维用户画像是为了帮你深入理解用户。但 PRD 里不需要那么详细。

❌ 过于详细：
```markdown
## 目标用户
小李，28岁，在互联网公司做运营，每天通勤1.5小时，
喜欢喝咖啡，周末会去健身房，有轻度焦虑症...
```

✅ 精简版：
```markdown
## 目标用户
职场人士，每天处理5-10件事务，需要简单的任务追踪工具
```

PRD 里的用户描述只需要保留**与产品相关的特征**。

## 本节核心要点

- ✅ PRD 的内容来自第二章的思考成果，不是凭空编写
- ✅ 先写"要解决的问题"，再写功能
- ✅ "不做清单"和"核心功能"同样重要
- ✅ 用户描述只保留与产品相关的特征，不需要完整画像
